✅ 架构亮点

复式记账架构 · 参考 Firefly III
余额由账本推导（single source of truth），转账生成配对交易，支持一键对账修正漂移。这是金融应用的核心正确性保障，架构决策非常专业。
Docker 多阶段构建 + GHCR 自动发布
生产镜像仅含运行时依赖，非 root 运行，内置健康检查。双模式 Compose（一体 / 外部 DB）覆盖主流 NAS 部署场景。
设计令牌系统 (Design Tokens)
OKLCH 色彩空间 + 完整语义令牌 + 亮/暗/系统三态主题，Geist 字体体系。tokens.css → styles.css → components.css 层次清晰。

⚠️ 架构问题

前端 app.js 单体化 (5,253 行)
所有业务管理器（AccountManager, TransactionManager, InvestmentManager 等 16+ 模块）集中在一个文件中。虽以对象字面量分区，但无 ES Module 拆分，维护成本高，多人协作困难。
js/app.js — 5,253 行
后端 routes.js 仍为重负载单体 (3,086 行)
已开始拆分为 routes/accounts.js、routes/auth.js、routes/ai.js，但转账、交易、预算、理财、统计等核心路由仍全部写在 routes.js 中。拆分进度仅 ~20%。
server/routes.js — 3,086 行 (已拆出 ~500 行到子模块)
双 package.json 冗余
根目录与 server/ 下各有一份 package.json，依赖清单重复维护。README 也承认"版本可能略有差异"。建议统一为单份，或明确根为开发用、server 为生产用。

✅ 流畅度亮点

命名规范统一 · 中英混用自然
变量/函数采用 camelCase，数据库字段 snake_case，CSS 变量 kebab-case，三套命名体系各自一致。业务术语中英对照清晰（如 reconcile = 对账）。
README 文档质量极高
900+ 行 README 包含完整 API 文档、部署指南、环境变量说明、复式记账规则。每个接口都有请求/响应示例，堪称项目文档范本。
统一响应结构
所有 API 统一 { success, data, message } 格式，错误码对照表完整。前端 api() 统一拦截处理，异常路径清晰。

⚠️ 流畅度问题

前端存在两套 api() 函数
app.js 内定义了自己的 api()（无 token 注入，依赖全局 header），js/api.js 又有独立的 ES Module api()。两者格式化函数(fmt) 实现不一致（api.js 用 toFixed，app.js 用 Intl.NumberFormat）。
js/app.js:14 vs js/api.js:5 — 两套并行 API 层
错误处理风格不完全统一
部分路由使用 handleServerError() 返回 500，部分直接用 res.status(400).json(fail(...))。转账余额不足返回 400，但按语义应为 422。错误码粒度需细化。
app.js 缺乏 Source Map / 编译步骤
原生 JS 直接运行，无构建步骤。5000+ 行单文件调试时错误栈定位困难。建议至少引入轻量 source map 或拆分为 ES Modules。

⚠️ 质量问题清单

[P0] app.js 5,253 行 — 严重违反单一职责
16 个 Manager 对象（AccountManager, TransactionManager, BudgetManager, InvestmentManager, DebtManager, DashboardManager, ReportManager 等）全部挤在一个全局作用域。虽有注释分区，但缺乏真正的模块隔离。任何修改都有引入回归的风险。
js/app.js — 建议拆分为 js/managers/*.js，每个模块独立 ES Module
[P0] 测试覆盖率极低 (~25%)
仅 3 个测试文件覆盖纯函数工具（validate, utils, debt-summary），核心的交易/转账/余额计算等关键业务路径零测试覆盖。smoke-test / full-verify 为手工脚本，未接入 CI。
test/ — 仅 3 个文件，无 API 集成测试
[P1] routes.js 3,086 行 — 路由拆分未完成
已开始拆分（accounts, auth, ai 已独立），但转账/交易/预算/理财/统计/报表/CSV/标签/储蓄目标等核心路由仍在一个文件中。按域拆分可显著提升可维护性。
server/routes.js — 建议继续拆分为 routes/transactions.js, routes/budgets.js 等
[P1] 数据库连接池未做优雅关闭
db.js 的 pool 创建后未见 pool.end() 调用。进程退出时（SIGTERM / SIGINT）应优雅关闭连接池，否则可能导致数据库连接泄漏或数据丢失。
server/db.js — 缺少 process.on('SIGTERM') 处理
[P1] SQL 查询拼接存在 XSS 间接风险
routes.js:341 中 tagIds 直接拼入 SQL IN 子句（`WHERE ... IN (${placeholders})`），虽然 placeholders 全是 '?'，但 ids 来自前端传入。当前实现安全（使用参数化），但模式易被误用。
[P2] ensureCategory 缺少 user_id 过滤
routes.js:222 的 ensureCategory 函数查询 categories 时未按 user_id 过滤（虽然插入时也无 user_id），可能导致多用户间分类冲突。categories 表 DDL 中也没有 user_id 列。
server/routes.js:221-234 + server/schema.sql:38-49
[P2] 理财行情代理使用外部数据源无缓存
天天基金/腾讯证券行情直接代理转发，无本地缓存层。频繁刷新会直接打到第三方 API，既影响性能也可能触发限流。建议至少加内存 TTL 缓存（如 5 分钟）。
[P2] index.html 905 行 — 单体 HTML 含全部页面
12 个 page section 全部内联在一个 HTML 文件中，通过 CSS display 切换。虽然避免了 SPA 路由的复杂性，但 DOM 节点庞大影响首屏性能。可考虑按需加载各页面。
index.html — 905 行

🔐 5. 安全性审查 优秀

✅ 密码 bcrypt 加盐哈希 (cost=10)
使用 bcryptjs，盐值 10 轮，无明文存储。生产环境强制校验 JWT_SECRET 不得为默认值。
✅ JWT 鉴权 + 登录频率限制
所有业务接口需 Bearer Token，按用户名+IP 组合限流（15min/5次），防暴力破解。
✅ 全参数化 SQL 查询
所有 SQL 均使用 ? 占位符参数化，无字符串拼接，杜绝 SQL 注入。
✅ Helmet CSP + XSS 防护
开启 Content-Security-Policy，前端用户可控字段统一 escapeHtml 转义。
✅ API Key AES-256-GCM 加密存储
AI 服务商的 API Key 使用 crypto.js 的 AES-256-GCM 加密后入库，非明文。
✅ 容器非 root + Docker 多阶段构建
运行镜像以 appuser 身份运行，COPY 仅含生产文件，攻击面最小化。
⚠️ CSP 允许 unsafe-inline script
因前端动态按钮使用内联 onclick，CSP 的 scriptSrc 包含 'unsafe-inline'。长期应改为 addEventListener 绑定以移除 inline handler。
ℹ️ MariaDB 3306 端口暴露到宿主机
docker-compose.yml 中 db 服务将 3306 映射到宿主机。生产 NAS 部署建议移除端口映射，仅通过内部网络访问。

🚀 7. 改进建议（按优先级排序）

🔴 高优先级 — 建议近期完成

  • 拆分 app.js：将 16 个 Manager 拆为独立 ES Module（js/managers/*.js），boot() 函数统一注册。预计可拆为 12-16 个 200-400 行的文件。
  • 补充集成测试：使用 Node.js 内置 test runner + supertest，覆盖交易 CRUD / 转账 / 余额计算等核心路径。目标覆盖率 > 70%。
  • 统一前端 api()：将 app.js 的 api() 迁移到 api.js ES Module，消除重复与行为差异。

🟡 中优先级 — 建议迭代推进

  • 继续拆分 routes.js：按域拆出 transactions.js, budgets.js, investments.js, stats.js, reports.js 等。目标单文件 < 500 行。
  • 添加 graceful shutdown：监听 SIGTERM/SIGINT，调用 pool.end() 后退出。
  • 行情缓存：引入内存 LRU 缓存（TTL 5 分钟），减少外部请求。
  • categories 增加 user_id：支持多用户独立分类体系。

🔵 低优先级 — 持续优化

  • 统一为单份 package.json，或明确双 manifest 的职责边界
  • CSP 移除 'unsafe-inline'，改用 addEventListener 绑定
  • index.html 按需加载页面（动态 fetch + inject section）
  • 移除 docker-compose.yml 中 db 的宿主机端口映射
  • 引入 ESLint + Prettier 统一代码风格

📝 审查结论

鑫钱包是一个功能完整、架构思路清晰的个人财务管理应用。复式记账模型参考 Firefly III，余额由账本推导的设计保证了数据正确性；Docker 多阶段构建 + GHCR 自动发布 + 双模式 Compose 的部署方案在同类个人项目中堪称标杆。安全层面（bcrypt + JWT + Helmet CSP + 参数化 SQL + AES-GCM 加密）也做到了应有尽有的防护。

主要短板集中在代码组织与测试覆盖两个维度：app.js 5,253 行和 routes.js 3,086 行的巨型文件严重影响可维护性；核心业务路径零测试覆盖意味着任何修改都缺乏安全网。建议优先完成前端模块拆分与集成测试补充，即可将整体质量从 A- 提升至 A 级。
